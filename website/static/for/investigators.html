For investigators

OpenExtract for investigators

Extract the content of an iPhone backup — encrypted included — with full metadata preserved. Free, open source, and runs 100% locally.

If you're a private investigator, a digital-forensics examiner, or an incident-response consultant, your relationship with iPhone data is probably hands-on. You've worked with Cellebrite, Magnet, iMazing Forensics, or a combination. You know what a good extraction looks like, and you know what questions a defensible examination has to answer.

OpenExtract isn't trying to replace a full forensic suite. It's a focused, free, open-source tool for the cases where the backup is what you have, a full forensic license isn't justified by the engagement, or the budget for a one-off matter doesn't cover a commercial tool.

What OpenExtract does well

  • Reads standard iTunes/Finder backups in the Apple manifest+MBDB format, including multi-device backup folders.
  • Decrypts password-protected backups given the password. Encrypted backups unlock significantly more artifacts — Safari history, health data, saved credentials, Wi-Fi networks, app-store history.
  • Reads every message the backup contains — which includes threads the user deleted from the device after a given backup was captured. OpenExtract does not do SQLite free-page carving or tombstone recovery; it reads the contents the backup has, straight.
  • Parses third-party app databases — WhatsApp, Signal, Telegram, and other apps that store chat data in SQLite. Output is structured, not screenshot-scraped.
  • Preserves EXIF on photos and videos — GPS coordinates, capture timestamps, device identifiers.
  • Exports to portable formats — PDF for reports, CSV for analysis, HTML for browsable archives, raw files for further processing in your toolchain.
  • Runs on Linux in addition to Windows and macOS, which matters if your examination workstation is hardened Linux.

What OpenExtract doesn't do

Being direct:

  • It is not a court-examiner suite. There is no built-in chain-of-custody report, no integrated hash-manifest against a source image, no write-blocker integration. If you need those, pair OpenExtract with your existing workflow (or continue using a commercial suite for matters that justify it).
  • It does no forensic recovery. No SQLite free-page carving, no tombstoned-row reconstruction, no chip-off or JTAG, no file-system carving. It reads what the backup contains. Anything that isn't in the backup is beyond its scope — take those cases to a vendor suite or a specialist examiner.
  • It doesn't pull from live devices. OpenExtract reads a backup that already exists on disk. It doesn't perform an AFC dump, a checkm8 extraction, or an advanced logical acquisition. Create the backup first (or obtain one), then extract.
  • It doesn't decrypt without the password. Encrypted backups require the password. OpenExtract does not attempt brute force, and integrating with password-recovery tools is out of scope by design.
  • It doesn't touch the cloud side. iCloud-held data that isn't in the backup — server-side message history for users with Messages-in-iCloud enabled, iCloud Photo Library above the on-device cache — is not extracted. Different path for that.
  • No vendor support contract. Issues are filed publicly on GitHub. If you need a vendor to sign an NDA and join a call, that's a different tool.

Why the open-source license is an advantage in this work

For examinations that end up in court, the open-source nature of OpenExtract is an asset:

  • Methodology is inspectable. If opposing counsel wants to understand exactly how a SQLite row is mapped to a rendered message, the answer is in the source. That's a different posture than "the vendor's extraction engine says so."
  • Reproducibility is easier to establish. The exact version is pinned by git commit; the build is reproducible from source.
  • No vendor-specific black box to explain. Your testimony doesn't depend on a vendor's willingness to support your report.

This doesn't replace the need for a qualified examiner. It gives the qualified examiner a more defensible posture.

Typical forensic workflow

  1. Obtain the iPhone backup. Either the subject creates it (unencrypted or encrypted — strongly prefer encrypted; more data is recovered), or you receive one from counsel with a preservation letter.
  2. Preserve the source. Hash the backup folder before you touch it. Store the source read-only. Work from a copy.
  3. Run OpenExtract on the copy. Load the backup folder; provide the password if encrypted.
  4. Export what's relevant. Message threads, call logs, attachments, app databases. PDF for the final report exhibits, CSV for internal analysis.
  5. Document the tool version. Record the OpenExtract release you used (visible in the app and in each export's header). Consider including the source-code commit if your jurisdiction or engagement calls for that level of specificity.
  6. Verify attachment integrity. OpenExtract exports attachment files separately from the rendered messages; hash them against the equivalents in the backup source if chain-of-custody matters.

Use cases we hear about

  • Background and due-diligence work — reviewing communications and location history on a device the subject consented to examine.
  • Internal corporate investigations — harassment, IP exfiltration, policy violations, with the employee's backup produced under a preservation letter.
  • Pre-litigation review — counsel commissioning a quiet review of a client's own device to understand what's there before committing to a case.
  • Missing-person and estate matters — recovering communications and photos from a device when the owner can't operate it.
  • Consumer incidents — helping a victim recover evidence of fraud, stalking, or harassment from their own device.
